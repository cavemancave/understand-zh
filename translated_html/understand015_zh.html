第十二章　共享内存虚拟文件系统

原文版本：Chapter 12 — Shared Memory Virtual Filesystem

共享一块由文件或设备作为后备的内存区，只需用 MAP_SHARED 标志调用 mmap() 即可。然而有两种重要情形是需要在进程间共享匿名区的：第一种是不带文件后备地以 MAP_SHARED 方式调用 mmap()，这类区域会在执行 fork() 之后由父子进程共享；第二种是通过 shmget() 显式创建一个区，然后用 shmat() 把它挂接到虚拟地址空间。

当 VMA 中的页由磁盘上的文件作为后备时，所用的接口很直接。要在缺页时读入一页，所需的 nopage() 函数可以通过 vm_area_struct→vm_ops 找到。要把一页写回后备存储，则在 address_space_operations 中查找相应的 writepage() 函数，通常经由 inode→i_mapping→a_ops，或者经由 page→mapping→a_ops。在进行 mmap()、read() 和 write() 等普通文件操作时，带有相应函数的 struct file_operations 通过 inode→i_fop 找到，等等。它们之间的关系在图 4.2 中已经展示过。

这是一种非常清晰、概念上很容易理解的接口，但它对匿名页并不适用，因为匿名页没有文件作为后备。为了沿用这套漂亮的接口，Linux 借助一个基于 RAM 的文件系统，为匿名页人造出文件后备：每一个 VMA 都对应该文件系统中的一个“文件”。该文件系统中的每个 inode 都挂在一个名为 shmem_inodes 的链表上，以便随时方便地定位它们。这样，同一套基于文件的接口便可被使用，而无需把匿名页当作特例处理。

这个文件系统有两个变体，分别叫做 shm 和 tmpfs。它们共享核心功能，主要区别在用途。shm 供内核为匿名页创建文件后备，以及为 shmget() 创建的共享区提供后备。该文件系统通过 kern_mount() 挂载，因此是在内部挂载、对用户不可见的。tmpfs 是一个临时文件系统，可以选择性地挂载到 /tmp/，提供一个基于 RAM 的高速临时文件系统。tmpfs 的另一个用途是挂载到 /dev/shm/：在该 tmpfs 文件系统中用 mmap() 映射文件的进程，可以以此在进程间共享信息，作为 System V IPC 机制的一种替代方案。无论用途如何，tmpfs 必须由系统管理员显式挂载。

本章先描述虚拟文件系统的实现方式，然后讨论共享区的建立与销毁，最后说明如何借助这些工具实现 System V IPC 机制。

12.1　初始化虚拟文件系统

虚拟文件系统由 init_tmpfs() 函数初始化，可能在系统启动时执行，也可能在模块被加载时执行。该函数注册 tmpfs 和 shm 两个文件系统，并用 kern_mount() 把 shm 挂载为内部文件系统。随后它计算文件系统中可以存在的最大块数与 inode 数。作为注册的一部分，shmem_read_super() 作为回调函数被用来填充 struct super_block，向其中写入文件系统的更多信息，例如把块大小设为页大小。

图 12.1：调用图：init_tmpfs()

该文件系统中创建的每个 inode 都会关联一个 struct shmem_inode_info，其中保存与本文件系统相关的私有信息。SHMEM_I() 函数以一个 inode 作为参数，返回指向该结构的指针。它在 <linux/shmem_fs.h> 中声明如下： 

 20 struct shmem_inode_info {
 21     spinlock_t              lock;
 22     unsigned long           next_index;
 23     swp_entry_t             i_direct[SHMEM_NR_DIRECT];
 24     void                  **i_indirect;
 25     unsigned long           swapped;
 26     unsigned long           flags;
 27     struct list_head        list;
 28     struct inode           *inode;
 29 };

各字段含义如下：

lock
保护 inode 信息免受并发访问的自旋锁。
next_index
文件中正在使用的最后一页的索引。文件被截断（truncate）期间，该值会与 inode→i_size 不同。
i_direct
直接块，包含该文件正在使用的前 SHMEM_NR_DIRECT 个交换向量。参见第 12.4.1 节。
i_indirect
指向第一个间接块的指针。参见第 12.4.1 节。
swapped
属于本文件、当前被换出到交换区的页数。
flags
目前仅用于记录该文件是否属于 shmget() 创建的共享区。通过 shmctl() 指定 SHM_LOCK 来置位，指定 SHM_UNLOCK 来解锁。
list
本文件系统所使用的全部 inode 的链表。
inode
指向其父 inode 的指针。

12.2　使用 shmem 相关函数

若干结构中都包含指向 shmem 专用函数的指针。所有这些场合，tmpfs 与 shm 共用同一组结构。

用于缺页换入页面以及把页面写回后备存储的，是两个名为 shmem_aops 和 shmem_vm_ops 的结构，分别属于 struct address_space_operations 与 struct vm_operations_struct。

地址空间操作 struct shmem_aops 包含指向若干函数的指针，其中最重要的是 shmem_writepage()，当一个页从页缓存被移到交换缓存时被调用。shmem_removepage() 在页从页缓存中被移除时调用，以便能回收对应的块。shmem_readpage() 不被 tmpfs 使用，但仍提供，以便 sendfile() 系统调用可用于 tmpfs 文件。shmem_prepare_write() 与 shmem_commit_write() 也未被使用，但提供它们是为了让 tmpfs 可以与回环设备（loopback device）配合使用。shmem_aops 在 mm/shmem.c 中声明如下： 

1500 static struct address_space_operations shmem_aops = {
1501     removepage:     shmem_removepage,
1502     writepage:      shmem_writepage,
1503 #ifdef CONFIG_TMPFS
1504     readpage:       shmem_readpage,
1505     prepare_write:  shmem_prepare_write,
1506     commit_write:   shmem_commit_write,
1507 #endif
1508 };

匿名 VMA 使用 shmem_vm_ops 作为其 vm_operations_struct，这样在缺页换入新页时就会调用 shmem_nopage()。它的声明如下： 

1426 static struct vm_operations_struct shmem_vm_ops = {
1427     nopage: shmem_nopage,
1428 };

要对文件和 inode 进行操作，需要 file_operations 与 inode_operations 这两个结构。file_operations 称为 shmem_file_operations，提供了实现 mmap()、read()、write() 与 fsync() 的函数，声明如下： 

1510 static struct file_operations shmem_file_operations = {
1511     mmap:           shmem_mmap,
1512 #ifdef CONFIG_TMPFS
1513     read:           shmem_file_read,
1514     write:          shmem_file_write,
1515     fsync:          shmem_sync_file,
1516 #endif
1517 };

共提供了三组 inode_operations。第一组是 shmem_inode_operations，用于文件 inode；第二组 shmem_dir_inode_operations 用于目录；最后一对 shmem_symlink_inline_operations 与 shmem_symlink_inode_operations 用于符号链接。

对文件 inode 支持的两种操作是 truncate() 与 setattr()，保存在名为 shmem_inode_operations 的 struct inode_operations 中。shmem_truncate() 用于截断文件。shmem_notify_change() 在文件属性发生变化时被调用，这使得（其他用途之外）可以通过 truncate() 增大一个文件、并以全局零页作为其数据页。shmem_inode_operations 的声明如下： 

1519 static struct inode_operations shmem_inode_operations = {
1520         truncate:       shmem_truncate,
1521         setattr:        shmem_notify_change,
1522 };

目录使用的 inode_operations 提供了诸如 create()、link() 与 mkdir() 等函数，声明如下： 

1524 static struct inode_operations shmem_dir_inode_operations = {
1525 #ifdef CONFIG_TMPFS
1526     create:         shmem_create,
1527     lookup:         shmem_lookup,
1528     link:           shmem_link,
1529     unlink:         shmem_unlink,
1530     symlink:        shmem_symlink,
1531     mkdir:          shmem_mkdir,
1532     rmdir:          shmem_rmdir,
1533     mknod:          shmem_mknod,
1534     rename:         shmem_rename,
1535 #endif
1536 };

最后一对操作用于符号链接，声明如下：

1354 static struct inode_operations shmem_symlink_inline_operations = {
1355         readlink:       shmem_readlink_inline,
1356         follow_link:    shmem_follow_link_inline,
1357 };
1358 
1359 static struct inode_operations shmem_symlink_inode_operations = {
1360         truncate:       shmem_truncate,
1361         readlink:       shmem_readlink,
1362         follow_link:    shmem_follow_link,
1363 };

两组 readlink() 与 follow_link() 函数的差别与链接信息的存放位置有关。符号链接 inode 并不需要私有 inode 信息 struct shmem_inode_information。如果符号链接的名字长度小于该结构的大小，就直接利用 inode 中的空间来保存名字，此时使用的是 shmem_symlink_inline_operations；否则会用 shmem_getpage() 分配一页，把符号链接内容复制到该页中，并改用 shmem_symlink_inode_operations。后一组结构包含 truncate() 函数，以便在文件被删除时回收该页。

这些结构合在一起，确保了当区域由虚拟文件作后备时，会使用 shmem 对应的 inode 相关操作。在它们投入使用之后，VM 的大部分代码就察觉不到由真实文件后备的页与由虚拟文件后备的页之间的差别了。

12.3　在 tmpfs 中创建文件

由于 tmpfs 被挂载为对用户可见的正规文件系统，它必须支持诸如 open()、mkdir() 和 link() 之类的目录 inode 操作。在 第 12.2 节展示过的 shmem_dir_inode_operations 中提供了实现这些操作的函数指针。

这些函数大多实现得相当简短，并且从某种程度上彼此联系紧密，图 12.2 即展示了这一点。它们都基于同一基本原则：对虚拟文件系统中的 inode 做一些工作，而 inode 的大部分字段是由 shmem_get_inode() 来填充的。

图 12.2：调用图：shmem_create()

创建新文件时，顶层调用的是 shmem_create()。这个小函数加上 S_IFREG 标志去调用 shmem_mknod()，从而创建一个普通文件。shmem_mknod() 本身只是对 shmem_get_inode() 的薄包装，后者如其名所示，会创建一个新 inode 并填充其结构字段。其中重点填充的三个字段是 inode→i_mapping→a_ops、inode→i_op 与 inode→i_fop。inode 创建好之后，shmem_mknod() 会更新目录 inode 的 size 与 mtime 等统计信息，然后实例化新 inode。

尽管两个文件系统在功能上本质相同，但在 shm 中文件的创建方式有所不同。shm 中文件的创建过程将在第 12.7 节讨论。

12.4　虚拟文件中的页错误处理

当发生缺页时，do_no_page() 会调用 vma→vm_ops→nopage（若存在）。对于虚拟文件系统而言，这意味着发生缺页时被调用的是 shmem_nopage()，其调用图见图 12.3。

图 12.3：调用图：shmem_nopage()

此处的核心函数是 shmem_getpage()，它负责分配一个新页，或在交换区中找到对应的页。这种把缺页类型重载的方式比较少见——通常应由 do_swap_page() 根据 PTE 中编码的信息来定位已经被移到交换缓存或后备存储中的页。但在这里，由虚拟文件后备的页在被移入交换缓存时，其 PTE 会被置 0；而该 inode 的文件系统私有数据保存着直接块和间接块信息，用于稍后定位页。这一过程在许多方面与普通缺页处理非常相似。

12.4.1　定位被换出的页

当一页被换出时，swp_entry_t 中会保存再次定位该页所需的信息。这一信息不是放在 PTE 中，而是保存在 inode 的文件系统私有信息里。

缺页时用于定位 swap 条目的函数是 shmem_alloc_entry()。它的基本职责是做一些基本检查，并确保 shmem_inode_info→next_index 始终指向虚拟文件末尾的页索引。它主要工作是调用 shmem_swp_entry()，在 inode 信息中查找交换向量；必要时还会分配新的页来存放交换向量。

前 SHMEM_NR_DIRECT 个条目存放在 inode→i_direct 中。这意味着在 x86 上，小于 64KiB（即 SHMEM_NR_DIRECT * PAGE_SIZE）的文件不需要使用间接块。更大的文件则必须使用间接块，从 inode→i_indirect 处的间接块开始。

图 12.4：在虚拟文件中遍历间接块

最初的间接块（inode→i_indirect）分为两半：前一半包含指向二级间接块的指针，后一半包含指向三级间接块的指针。二级间接块本身是包含交换向量（swp_entry_t）的页。三级间接块包含指向其他页的指针，这些页中又填满了交换向量。不同层级的间接块之间的关系见图 12.4。由此关系可知，虚拟文件中最大页数（SHMEM_MAX_INDEX）在 mm/shmem.c 中定义如下： 

 44 #define SHMEM_MAX_INDEX  (
         SHMEM_NR_DIRECT + 
         (ENTRIES_PER_PAGEPAGE/2) *
         (ENTRIES_PER_PAGE+1))

12.4.2　把页写入交换区

shmem_writepage() 是在该文件系统的 address_space_operations 中登记的、用于把页写入交换区的函数。该函数所做的只是把页从页缓存移到交换缓存，分几个简单步骤实现：

  • 记录当前的 page→mapping 以及关于 inode 的信息；
  • 用 get_swap_page() 在后备存储中分配一个空闲槽位；
  • 用 shmem_swp_entry() 分配一个 swp_entry_t；
  • 把该页从页缓存中移除；
  • 把该页加入交换缓存。如果失败，则释放该交换槽位，把页放回页缓存中，并重试。

12.5　tmpfs 中的文件操作

虚拟文件支持四种操作：mmap()、read()、write() 与 fsync()，函数指针保存在 第 12.2 节展示过的 shmem_file_operations 中。

这些操作的实现没有什么特别之处，其细节会在配套的代码注释（Code Commentary）中给出。mmap() 操作由 shmem_mmap() 实现，仅仅更新管理被映射区的 VMA。read() 由 shmem_read() 实现，它把虚拟文件中的字节复制到用户态缓冲区，必要时缺页换入页面。write() 由 shmem_write() 实现，基本与之相同。fsync() 由 shmem_file_sync() 实现，但实际上是一个空操作——它不做任何事，直接返回 0 表示成功。因为文件只存在于 RAM 中，根本不需要与磁盘同步。

12.6　tmpfs 中的 inode 操作

inode 上支持的最复杂操作是截断（truncation），它涉及四个明显的阶段。第一阶段在 shmem_truncate() 中，会先截断文件末尾不完整的那一页，并不断调用 shmem_truncate_indirect()，直到文件被截断到目标大小。每次调用 shmem_truncate_indirect() 一趟只处理一个间接块，因此可能需要被多次调用。

第二阶段在 shmem_truncate_indirect() 中，能够处理二级和三级间接块，它会找到下一个需要被截断的间接块。这个间接块会被传给第三阶段，它包含指向若干页的指针，而这些页又包含交换向量。

第三阶段在 shmem_truncate_direct() 中，处理包含交换向量的页。它选出需要被截断的范围，然后把这一范围交给最后一个阶段 shmem_swp_free()。最后阶段用 free_swap_and_cache() 释放各个条目，该函数会同时释放交换条目和包含数据的页。

文件的链接与解链接非常简单，因为大部分工作由文件系统层完成。链接一个文件时，目录 inode 的大小递增，相关 inode 的 ctime 与 mtime 被更新，被链接到的 inode 的链接计数也递增。然后用 dget() 取得对新 dentry 的引用，再用 d_instantiate() 实例化新的 dentry。解链接时同样会更新这些 inode 统计信息，并用 dput() 减少对 dentry 的引用。dput() 也会调用 iput()，当 inode 引用计数归零时清理该 inode。

创建目录会用 shmem_mkdir() 来完成。它只是带上 S_IFDIR 标志调用 shmem_mknod()，然后把父目录 inode 的 i_nlink 计数递增。shmem_rmdir() 用于删除目录，它先通过 shmem_empty() 确保目录为空。若为空，则把父目录 inode 的 i_nlink 减一，并调用 shmem_unlink() 删除该目录。

12.7　建立共享区

共享区由一个在 shm 中创建的文件作为后备。有两种情形会创建新文件：用 shmget() 建立共享区时，以及用带 MAP_SHARED 标志的 mmap() 建立匿名区时。两种情形都使用核心函数 shmem_file_setup() 来创建文件。

图 12.5：调用图：shmem_zero_setup()

由于该文件系统是内部的，所创建的文件名不必唯一，因为文件总是按 inode 而非按名字定位。因此 shmem_zero_setup() 总是创建一个名为 dev/zero 的文件——它就以这个名字出现在 /proc/pid/maps 中。而由 shmget() 创建的文件名为 SYSVNN，其中 NN 是传给 shmget() 的 key 参数。

核心函数 shmem_file_setup() 只是创建一个新 dentry 和 inode，填充相关字段，然后将其实例化。

12.8　System V IPC

IPC 的全部内部实现细节超出了本书的范围。本节只关注 shmget() 与 shmat() 的实现，以及它们如何受到 VM 的影响。系统调用 shmget() 由 sys_shmget() 实现。它对参数做基本检查，并建立 IPC 相关的数据结构。为创建段，它会调用 newseg()。如上一节所述，newseg() 正是借助 shmem_file_setup() 在 shmfs 中创建文件的函数。

图 12.6：调用图：sys_shmget()

系统调用 shmat() 由 sys_shmat() 实现，其本身没什么特别值得一提的内容。它获取相应的描述符，确认全部参数都合法，然后调用 do_mmap()，把共享区映射到进程的地址空间。该函数只有两点值得注意。

第一，若调用者指定了地址，则该函数负责确保 VMA 不会相互重叠。第二，shp→shm_nattch 计数器由一个名为 shm_vm_ops 的 vm_operations_struct 来维护，它注册了两个回调 shm_open() 与 shm_close()，分别对应 open() 与 close()。shm_close() 还负责在指定了 SHM_DEST 标志且 shm_nattch 归零时销毁共享区。

12.9　2.6 中的新变化

该文件系统的核心概念与功能保持不变，所做的修改要么是优化，要么是对功能的扩展。如果读者对 2.4 的实现已经理解透彻，则 2.6 的实现应不会带来太大困难1。

在 shmem_inode_info 中新增了一个字段 alloced。alloced 字段保存文件已分配了多少数据页——在 2.4 中，这一数值是基于 inode→i_blocks 临时算出来的。该字段既在常见操作上节省了一些时钟周期，也使代码更易读。

现在 flags 字段除了 VM_LOCKED 之外还使用 VM_ACCOUNT 标志。VM_ACCOUNT 始终被置位，表示 VM 会精细地核算所用的内存量，以确保分配不会失败。

文件操作的扩展包括：可通过 _llseek() 系统调用进行 seek（由 generic_file_llseek() 实现），以及对虚拟文件使用 sendfile()（由 shmem_file_sendfile() 实现）。VMA 操作也新增了一项扩展，以支持非线性映射，由 shmem_populate() 实现。

最后一个重要变化是：文件系统自身负责其 inode 的分配与销毁，对应的是 struct super_operations 中的两个新回调。其实现很简单——创建一个名为 shmem_inode_cache 的 slab 缓存。同时为 slab 分配器注册一个构造函数 init_once()，用于初始化每个新 inode。
