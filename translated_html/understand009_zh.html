第六章　物理页面分配

原文版本：Chapter 6 — Physical Page Allocation

本章介绍Linux中物理页面的管理与分配机制。所采用的核心算法是二进制伙伴分配器（Binary Buddy Allocator），由Knowlton [Kno65] 提出，并由Knuth [Knu68] 进一步描述。与其他分配器相比，该算法被证明具有极高的效率 [KB85]。

该分配方案结合了普通的2的幂次分配器与空闲缓冲区合并机制 [Vah96]，其基本思想非常简洁：内存被划分为若干大块页面，每块的大小是2的幂次个页面。若没有所需大小的空闲块，则将一个更大的块一分为二，形成两个伙伴块，一半用于分配，另一半放入空闲链表。如此不断折半，直到找到合适大小的块。当某块后来被释放时，会检查其伙伴块，若伙伴也空闲则合并成更大的块。

本章首先介绍Linux如何记录哪些内存块是空闲的，然后详细讨论分配和释放页面的方法，接着介绍影响分配器行为的标志位，最后讨论碎片问题以及分配器的应对策略。

6.1　管理空闲块

如前所述，分配器维护着若干空闲页面块，每块的大小是2的幂次个页面。2的幂次方的指数称为阶（order）。系统为每个阶维护一个 free_area_t 结构体数组，每个元素指向一个相应大小的空闲页面块链表，如图 6.1所示。

图 6.1：空闲页面块管理（每个 zone 的 free_area 数组与同阶空闲链表）

因此，数组第0个元素指向大小为 20=1 个页面的空闲块链表，第1个元素为 21=2 个页面的链表，依此类推，直到 2MAX_ORDER−1 个页面（其中 MAX_ORDER 当前定义为10）。这确保了不会为了满足小请求而拆分大块。页面块通过 page→list 以线性链表方式维护。

每个zone都有一个名为 free_area[MAX_ORDER] 的 free_area_t 结构数组，在 <linux/mm.h> 中声明如下： 

 22 typedef struct free_area_struct {
 23         struct list_head        free_list;
 24         unsigned long           *map;
 25 } free_area_t;

该结构体的字段含义如下：

  • free_list：空闲页面块的链表；
  • map：一个位图，表示一对伙伴块的状态。

Linux为了节省内存，对每对伙伴块只使用一个比特（而非两个）来表示状态。每当某个伙伴被分配或释放时，代表该对伙伴的比特就会翻转：若该位为0，表示两个伙伴都空闲或都已分配；若为1，表示只有一个伙伴在使用中。page_alloc.c 中的宏 MARK_USED() 用于翻转对应比特，定义如下： 

164 #define MARK_USED(index, order, area) \
165         __change_bit((index) >> (1+(order)), (area)->map)

index 是页面在全局 mem_map 数组中的索引。将其右移 1+order 位，即可定位到 map 中代表该对伙伴的比特位。

6.2　分配页面

Linux提供了一套相当完整的页帧分配API，所有函数都接受一个 gfp_mask 参数，该参数是一组决定分配器行为的标志位（详见第6.4节）。

所有分配API函数最终都调用核心函数 __alloc_pages()，但各API的存在是为了让调用者能自动选择正确的内存节点和zone（如 ZONE_DMA、ZONE_NORMAL 等），调用者无需关心具体使用哪个节点。物理页面分配API完整列表见表 6.1。

表 6.1：物理页面分配API

函数原型说明
struct page * alloc_page(unsigned int gfp_mask) 分配单个页面，返回 struct page 指针
struct page * alloc_pages(unsigned int gfp_mask, unsigned int order) 分配 2order 个页面，返回 struct page 指针
unsigned long get_free_page(unsigned int gfp_mask) 分配单个页面，清零后返回虚拟地址
unsigned long __get_free_page(unsigned int gfp_mask) 分配单个页面，返回虚拟地址
unsigned long __get_free_pages(unsigned int gfp_mask, unsigned int order) 分配 2order 个页面，返回虚拟地址
struct page * __get_dma_pages(unsigned int gfp_mask, unsigned int order) 从DMA zone分配 2order 个页面，返回 struct page 指针

分配总是按指定阶进行（分配单页时阶为0）。若找不到所需阶的空闲块，则将高阶块一分为二：一半用于分配，另一半加入低阶空闲链表。图 6.2展示了一个24阶块的拆分过程及伙伴块如何逐步加入空闲链表，直到找到满足请求的块。

图 6.2：物理页面分配（伙伴系统拆分 2⁴ 块以满足 2¹ 请求；order 自上而下从小到大，时间从下往上）

当块被释放时，会检查其伙伴：若两者均空闲，则合并成高阶块并加入高阶空闲链表，并继续检查更高阶的伙伴；若伙伴不空闲，则将释放的块加入当前阶的空闲链表。在这些链表操作期间，需要禁用中断，以防中断处理程序在链表处于不一致状态时操作它，这通过中断安全的自旋锁实现。

第二个决策是选择哪个内存节点（pg_data_t）。Linux采用本地节点分配策略，优先使用与当前运行进程的CPU相关联的内存。这里关键函数是 _alloc_pages()，该函数在UMA架构（mm/page_alloc.c）和NUMA架构（mm/numa.c）下有不同的实现。

无论使用哪种API，mm/page_alloc.c 中的 __alloc_pages() 都是分配器的核心。该函数不应被直接调用，它检查所选zone是否适合分配（根据可用页面数量），若不适合则可能回退到其他zone（启动时由 build_zonelists() 决定回退顺序，通常 ZONE_HIGHMEM 回退到 ZONE_NORMAL，再回退到 ZONE_DMA）。若空闲页面数降至 pages_low 水位线，则唤醒 kswapd 开始释放页面；若内存极度紧张，调用者将自行执行 kswapd 的工作。

图 6.3：调用图：alloc_pages()

一旦确定了zone，就会调用 rmqueue() 分配页面块，或在没有合适大小块时拆分高阶块。

6.3　释放页面

页面释放的API相对简单，其存在是为了便于记住释放时块的阶——伙伴分配器的一个缺点是调用者必须记住原始分配的大小。释放API见表 6.2。

表 6.2：物理页面释放API

函数原型说明
void __free_pages(struct page *page, unsigned int order) 从给定页面释放指定阶数量的页面
void __free_page(struct page *page) 释放单个页面
void free_page(void *addr) 从给定虚拟地址释放页面

释放页面的核心函数是 __free_pages_ok()，不应直接调用。而应使用 __free_pages()，它会先进行一些基本检查，如图 6.4所示。

图 6.4：调用图：__free_pages()（粉色虚框为伙伴合并循环）

当伙伴被释放时，Linux会立即尝试合并伙伴块（如果可能）。这并非最优策略，因为最坏情况下会出现大量合并后立即再次拆分的情况 [Vah96]。

为了检测伙伴是否可以合并，Linux检查 free_area→map 中对应的比特位。由于一个伙伴刚被释放，至少知道一个是空闲的。翻转后若该位为0，则说明另一个伙伴也是空闲的（位为0意味着两者都空闲或都已分配），此时可以合并。

计算伙伴地址是一个经典算法 [Knu68]。由于分配总是按 2k 大小的块进行，块的地址（至少是其在 zone_mem_map 内的偏移）也是 2k 的幂次，因此地址右边至少有 k 个零。为了得到伙伴的地址，检查地址右边第 k 位，若为0则翻转该位。Linux通过以下方式计算掩码： 

mask = (~0 << k)
imask = 1 + ~mask

Linux利用如下等式简化计算：

imask = -mask = 1 + ~mask

合并后，从空闲链表中移除该块，新合并的块移至下一阶，继续检查是否可进一步合并。

6.4　获取空闲页（GFP）标志

获取空闲页（GFP）标志是贯穿整个虚拟内存管理的核心概念。这些标志决定了分配器和 kswapd 在分配和释放页面时的行为。例如，中断处理程序不能休眠，因此不会设置 __GFP_WAIT 标志（该标志表示调用者可以休眠）。所有GFP标志均定义在 <linux/mm.h> 中，分为三组。

zone修饰符标志

第一组是zone修饰符，如表 6.3所示。这些标志指示调用者必须尝试从特定zone分配内存。注意没有 ZONE_NORMAL 的修饰符标志，因为修饰符标志用作数组偏移量，0隐式表示从 ZONE_NORMAL 分配。

表 6.3：影响zone分配的低级GFP标志

标志说明
__GFP_DMA尽可能从 ZONE_DMA 分配
__GFP_HIGHMEM尽可能从 ZONE_HIGHMEM 分配
GFP_DMA__GFP_DMA 的别名

行为修饰符标志

第二组是行为修饰符，如表 6.4所示，它们改变VM的行为和调用进程的权限。低级标志单独使用过于原始，不便直接操作。

__GFP_WAIT 表示调用者非高优先级，可以休眠或重新调度

6.5　避免碎片化

任何分配器都必须解决的重要问题是内部碎片和外部碎片。外部碎片是指可用内存只存在于小块中，无法满足大块请求。内部碎片是指为了满足小请求而分配大块，导致空间浪费。

在Linux中，外部碎片并不严重，因为对连续大页面的请求很少见，通常 vmalloc()（见第7章）已足够。空闲块链表确保了大块不会被不必要地拆分。

内部碎片是二进制伙伴系统最严重的缺陷。研究表明碎片率预期在28%左右 [WJNB95]，但实际可高达60%，而首次适应分配器仅约1% [JW98]。使用伙伴系统的变体也不能显著改善这种情况 [PN77]。

为解决此问题，Linux使用slab分配器 [Bon94] 将页面切分为小块内存进行分配 [Tan01]，详见第8章。通过这两种分配器的结合，内核能确保因内部碎片造成的内存浪费降到最低。

6.6　2.6 版本的新特性

页面分配

首个显著差异看似是表面上的改动：alloc_pages() 现在是宏，定义在 <linux/gfp.h> 而非 <linux/mm.h> 中的函数。新布局依然清晰，主要差异微妙但重要：2.4版本有专门代码用于根据当前CPU选择分配节点以区分NUMA和UMA架构，而2.6版本消除了这一区分。

在2.6中，alloc_pages() 调用 numa_node_id() 返回当前运行CPU关联节点的逻辑ID（NID），并传递给 _alloc_pages()，后者以NID为参数调用 NODE_DATA()。在UMA架构上，这无条件返回 contig_page_data；NUMA架构则以NID为数组偏移建立映射。本质上仍是与2.4相同的本地节点分配策略，但定义更清晰。

每CPU页面列表

页面分配最重要的新增是per-CPU列表（首次在第2.6节描述物理内存时提及）。

在2.4中，页面分配需要持有中断安全自旋锁。在2.6中，页面通过 buffered_rmqueue() 从 struct per_cpu_pageset 分配。若未达到低水位线（per_cpu_pageset→low），则无需持有自旋锁即可从pageset分配。达到低水位线后，将在持有中断安全自旋锁的情况下批量分配大量页面，加入per-cpu列表，再返回一个给调用者。

高阶分配（相对较少）仍需持有中断安全自旋锁，拆分和合并不会延迟。对于0阶分配，拆分会延迟到per-cpu集合的低水位线，合并会延迟到高水位线。

严格来说，这并不是惰性伙伴算法 [BL89]。pageset引入了0阶分配的合并延迟，这是副作用而非有意为之，且没有机制来排空pageset并合并伙伴块。尽管per-cpu和新的计账代码大幅增加了 mm/page_alloc.c 的代码量，伙伴算法的核心与2.4相比并无变化。

此改变的意义显而易见：减少了保护伙伴链表的自旋锁获取次数。高阶分配在Linux中相对较少，因此优化针对常见情况。此改变在多CPU机器上效果明显，对单CPU影响不大。pageset存在一些问题但不被认为是严重问题：

  • 若pageset持有通常会合并为高阶连续块的0阶页面，高阶分配可能失败；
  • 若内存不足、当前CPU的pageset为空而其他CPU的pageset已满，0阶分配可能失败（无机制从"远程"pageset回收页面）；
  • 新释放页面的伙伴可能在其他pageset中，导致潜在的碎片问题。
释放页面

引入了两个新的释放API：free_hot_page() 和 free_cold_page()，分别将释放的页面放入per-cpu pageset的热页或冷页列表。但 free_cold_page() 虽然被导出，实际上从未被调用。

来自 __free_pages() 的0阶页面释放以及 __page_cache_release() 引起的页缓存释放被放入热列表，而高阶分配则通过 __free_pages_ok() 立即释放。0阶分配通常与用户空间相关，是最常见的分配和释放类型，将它们保持在本地CPU可以减少锁竞争。

最终，页面列表必须传递给 free_pages_bulk()，否则pageset列表会持有所有空闲页面。该函数接受页面块链表、每块的阶和从链表中释放的块数量。主要有两种使用场景：高阶释放传递给 __free_pages_ok()（页面块以指定阶放入链表，count为1）；以及当运行CPU的pageset达到高水位线时（以阶0和count=pageset→batch传递pageset）。

一旦到达核心函数 __free_pages_bulk()，释放页面到伙伴链表的机制与2.4非常相似。

GFP 标志

仍然只有三个zone，zone修饰符保持不变，但新增了三个GFP标志，影响VM在满足请求时的努力程度：

  • __GFP_NOFAIL：调用者要求分配永不失败，分配器应无限重试；
  • __GFP_REPEAT：失败时应重试分配。当前实现与 __GFP_NOFAIL 行为相同，未来可能在一段时间后允许失败；
  • __GFP_NORETRY：与 __GFP_NOFAIL 几乎相反，分配失败时应立即返回。

撰写本文时，这些标志使用还不多，但已被引入并可能随时间增加使用量。__GFP_REPEAT 标志尤其可能被大量使用，因为实现其行为的代码块已遍布内核各处。

新引入的分配修饰符 __GFP_COLD 用于确保从per-cpu列表的冷页中分配。VM层面，唯一使用此标志的是 page_cache_alloc_cold()，主要在IO预读时使用。通常页面分配会从热页列表取。

最后一个新标志是 __GFP_NO_GROW，这是仅由slab分配器（第8章）使用的内部标志，是 SLAB_NO_GROW 标志的别名，用于指示永不为特定缓存分配新slab。实际上，引入此GFP标志只是为了补充目前在主内核中未使用的旧 SLAB_NO_GROW 标志。
