第二章　描述物理内存

原文版本：Chapter 2 — Describing Physical Memory

Linux 可以运行在多种不同的硬件架构上，因此需要一种与架构无关的方式来描述内存。本章将介绍内核用于记录内存块（bank）、物理页以及影响 VM 行为的标志位所使用的数据结构。

VM 中第一个普遍存在的核心概念是非一致性内存访问（Non-Uniform Memory Access，NUMA）。在大型机器上，内存可能被组织成若干 bank，处理器访问它们的开销会随着到 CPU 的"距离"不同而不同。例如，可能有一组内存专门分配给每个 CPU，或者有一组内存非常适合放在外设网卡附近用于 DMA。

每个 bank 称为一个结点（node），在 Linux 中由 struct pglist_data 表示——即使在 UMA 架构上也是如此。该结构总是通过其 typedef pg_data_t 来引用。系统中所有结点都挂在一个以 NULL 结尾的链表 pgdat_list 上，每个结点通过 pg_data_t->node_next 字段链接到下一个结点。对于像 PC 桌面这样的 UMA 架构，只使用一个静态的 pg_data_t 结构，称为 contig_page_data。结点的更多内容将在 2.1 节进一步讨论。

每个结点又被划分为若干称为区（zone）的块，每个区代表内存中的某段范围。注意，这里的 zone 与"基于 zone 的分配器（zone based allocator）"中的 zone 含义不同，两者并无关联。区由 struct zone_struct 描述，typedef 为 zone_t，类型为 ZONE_DMA、ZONE_NORMAL 或 ZONE_HIGHMEM。每种类型对应不同的用途：ZONE_DMA 是较低物理内存范围，某些 ISA 设备需要使用；ZONE_NORMAL 中的内存由内核直接映射到线性地址空间的上部区域（详见 4.1 节）；ZONE_HIGHMEM 是系统中剩余可用的内存，内核不直接映射。

在 x86 架构上，各 zone 的范围为：

ZONE_DMA低端的前 16 MiB 内存
ZONE_NORMAL16 MiB – 896 MiB
ZONE_HIGHMEM896 MiB 直至结束

需要特别注意的是，许多内核操作只能在 ZONE_NORMAL 上执行，因此它是性能最关键的区。Zone 的更多内容将在 2.2 节讨论。每个物理页帧都由一个 struct page 表示，所有 page 结构保存在一个全局数组 mem_map 中，它通常位于 ZONE_NORMAL 的起始处，或在低端内存机器上紧接在已加载的内核映像保留区之后。struct page 详见 2.4 节，全局 mem_map 数组详见 3.7 节。这些结构之间的基本关系如图 2.1 所示。

由于内核可直接访问的内存（ZONE_NORMAL）大小有限，Linux 引入了高端内存（High Memory）的概念，详见 2.5 节。本章将先介绍结点、区与页的表示，然后再引入高端内存管理。

2.1　结点（Nodes）

如前所述，每个内存结点由 pg_data_t（即 struct pglist_data）描述。在分配页面时，Linux 采用本地结点分配策略（node-local allocation policy），从距离当前运行 CPU 最近的结点分配内存。由于进程倾向于在同一 CPU 上运行，因此通常会使用当前结点的内存。该结构在 <linux/mmzone.h> 中声明如下：

129 typedef struct pglist_data {
130     zone_t node_zones[MAX_NR_ZONES];
131     zonelist_t node_zonelists[GFP_ZONEMASK+1];
132     int nr_zones;
133     struct page *node_mem_map;
134     unsigned long *valid_addr_bitmap;
135     struct bootmem_data *bdata;
136     unsigned long node_start_paddr;
137     unsigned long node_start_mapnr;
138     unsigned long node_size;
139     int node_id;
140     struct pglist_data *node_next;
141 } pg_data_t;

下面简要说明每个字段：

  • node_zones　本结点的 zone，依次为 ZONE_HIGHMEM、ZONE_NORMAL、ZONE_DMA；
  • node_zonelists　分配时偏好的 zone 顺序。该顺序由 mm/page_alloc.c 中的 build_zonelists() 在 free_area_init_core() 调用时建立。在 ZONE_HIGHMEM 分配失败时可能回退到 ZONE_NORMAL，进一步失败则回退到 ZONE_DMA；
  • nr_zones　本结点中 zone 的数量，介于 1 到 3 之间。并非所有结点都有 3 个 zone，例如某个 CPU bank 可能没有 ZONE_DMA；
  • node_mem_map　代表本结点中每个物理帧的 struct page 数组的首页，位于全局 mem_map 数组中的某处；
  • valid_addr_bitmap　用于描述结点内存中没有实际内存对应的"空洞"。实际上只有 Sparc 与 Sparc64 架构使用它，其它架构忽略；
  • bdata　仅对启动内存分配器（详见第 5 章）有意义；
  • node_start_paddr　结点的起始物理地址。使用 unsigned long 并不理想，因为它对启用 PAE 的 ia32 等架构来说不够用（PAE 见 2.5 节）。更合适的做法是使用页帧号（Page Frame Number，PFN）。PFN 就是按页大小为单位、在物理内存中的索引。某物理地址的 PFN 可以简单定义为 (page_phys_addr >> PAGE_SHIFT)；
  • node_start_mapnr　给出本结点在全局 mem_map 中的页面偏移。由 free_area_init_core() 计算，等于 mem_map 与本结点局部 mem_map（即 lmem_map）之间的页面数；
  • node_size　本 zone 中的页面总数；
  • node_id　结点 ID（NID），从 0 开始；
  • node_next　指向 NULL 结尾链表中的下一个结点。

系统中所有结点都挂在 pgdat_list 链表上。结点是在被 init_bootmem_core() 初始化时挂入该链表的（见 5.2.1 节）。在 2.4.18 之前的内核中，遍历该链表的典型代码为：

    pg_data_t * pgdat;
    pgdat = pgdat_list;
    do {
          /* 对 pgdata_t 做点什么 */
          ...
    } while ((pgdat = pgdat->node_next));

较新的内核提供了宏 for_each_pgdat()，本质上是一个 for 循环，用以提高代码可读性。

2.2　Zone

Zone 由 struct zone_struct 描述，通常通过 typedef zone_t 引用。它记录页面使用统计、空闲区域信息和锁等。声明位于 <linux/mmzone.h>：

37 typedef struct zone_struct {
41     spinlock_t        lock;
42     unsigned long     free_pages;
43     unsigned long     pages_min, pages_low, pages_high;
44     int               need_balance;
45
49     free_area_t       free_area[MAX_ORDER];
50
76     wait_queue_head_t * wait_table;
77     unsigned long     wait_table_size;
78     unsigned long     wait_table_shift;
79
83     struct pglist_data *zone_pgdat;
84     struct page        *zone_mem_map;
85     unsigned long      zone_start_paddr;
86     unsigned long      zone_start_mapnr;
87
91     char               *name;
92     unsigned long      size;
93 } zone_t;

各字段简要说明：

  • lock　自旋锁，保护本 zone 不被并发访问；
  • free_pages　zone 中的空闲页总数；
  • pages_min, pages_low, pages_high　zone 的三条水位线（watermark），将在下一节介绍；
  • need_balance　通知页换出进程 kswapd 对本 zone 进行平衡的标志。当可用页数达到某条水位线时，称该 zone "需要平衡"；
  • free_area　伙伴系统（buddy allocator）使用的空闲区位图；
  • wait_table　等待某页被释放的进程的等待队列哈希表。它对 wait_on_page() 和 unlock_page() 很重要。如果所有进程都等在同一个队列上，被唤醒时它们就会争抢仍被锁住的其它页面，这种现象称为"惊群（thundering herd）"。等待表详见 2.2.3 节；
  • wait_table_size　哈希表中队列的数量，是 2 的幂；
  • wait_table_shift　long 类型的位数减去 wait_table_size 的二进制对数；
  • zone_pgdat　指向父 pg_data_t；
  • zone_mem_map　本 zone 在全局 mem_map 中对应的首页；
  • zone_start_paddr　与 node_start_paddr 含义相同；
  • zone_start_mapnr　与 node_start_mapnr 含义相同；
  • name　zone 的字符串名："DMA"、"Normal" 或 "HighMem"；
  • size　以页为单位的 zone 大小。

2.2.1　Zone 水位线（Watermarks）

当系统可用内存较少时，页换出守护进程 kswapd 会被唤醒以开始释放页面（见第 10 章）。如果压力很大，分配进程本身会同步释放内存，这条路径有时称为直接回收（direct-reclaim）。这些影响换页行为的参数与 FreeBSD [McK96] 和 Solaris [MM01] 类似。

每个 zone 有三条水位线：pages_low、pages_min 和 pages_high，用于跟踪 zone 承受的压力。它们之间的关系如图 2.2 所示。pages_min 的值在内存初始化时由 free_area_init_core() 计算，与 zone 的页数大小成比例，初值为 ZoneSizeInPages / 128。其最小值为 20 页（在 x86 上约 80 KiB），最大值为 255 页（在 x86 上约 1 MiB）。

图 2.2：Zone 水位线
  • pages_low　当空闲页数达到 pages_low 时，伙伴分配器会唤醒 kswapd 开始释放页面。这相当于 Solaris 中的 lotsfree，FreeBSD 中的 freemin。其默认值是 pages_min 的两倍；
  • pages_min　当达到 pages_min 时，分配器将同步执行 kswapd 的工作，即直接回收路径。Solaris 中没有完全等价的概念，最接近的是 desfree 或 minfree，它们决定换页扫描进程的唤醒频率；
  • pages_high　一旦 kswapd 被唤醒开始释放页面，只有当空闲页数达到 pages_high 时才认为 zone 已"平衡"，此后 kswapd 会重新进入睡眠。Solaris 中称为 lotsfree，BSD 中称为 free_target。默认为 pages_min 的三倍。

无论各操作系统对换页参数的命名如何，其含义都是一样的：决定换页守护进程或其它进程释放页面的努力程度。

2.2.2　计算 Zone 的大小

图 2.3：函数调用图：setup_memory()

PFN 是按页计数的物理内存映射中的偏移。系统可用的第一个 PFN，即 min_low_pfn，位于 _end（已加载内核映像的末尾）之后第一个页面的开头。该值作为文件作用域变量保存在 mm/bootmem.c 中，供启动内存分配器使用。

系统中的最后一个页帧 max_pfn 的计算方式与架构密切相关。在 x86 上，函数 find_max_pfn() 遍历整个 e820 映射表查找最高的页帧。e820 是 BIOS 提供的一张表，描述哪些物理内存可用、被保留或不存在。该值同样作为文件作用域变量保存在 mm/bootmem.c。

max_low_pfn 在 x86 上由 find_max_low_pfn() 计算，标志 ZONE_NORMAL 的末尾。它代表内核可直接访问的物理内存，与线性地址空间中由 PAGE_OFFSET 划分的内核/用户空间相关。注意，在低端内存机器上 max_pfn 与 max_low_pfn 相等。

有了 min_low_pfn、max_low_pfn 和 max_pfn 这三个变量，就能很容易地算出高端内存的起止，并保存在 arch/i386/mm/init.c 的文件作用域变量 highstart_pfn 与 highend_pfn 中。这些值稍后将用于初始化高端内存页面以供物理页分配器使用（见 5.5 节）。

2.2.3　Zone 等待队列表

对页面执行 IO（换入或换出）时，会将其加锁以防止访问到不一致的数据。希望使用该页的进程必须先调用 wait_on_page() 加入等待队列。IO 完成后页面通过 UnlockPage() 解锁，等待队列上的所有进程会被唤醒。理论上每个页面都可以有自己的等待队列，但成本太高，因此 Linux 把等待队列存放在 zone_t 中。

整个 zone 也可以只用一个等待队列，但这意味着只要 zone 中任一页解锁，所有在该 zone 中等待的进程都会被唤醒，造成严重的惊群（thundering herd）问题。因此 zone_t->wait_table 存放的是一张等待队列哈希表。哈希冲突时可能仍会无谓地唤醒进程，但冲突预计不会频繁出现。

图 2.4：在锁住的页上等待

等待表在 free_area_init_core() 中分配。表的大小由 wait_table_size() 计算并保存在 zone_t->wait_table_size 中，上限为 4096 个等待队列。对于较小的表，大小为足以容纳 NoPages / PAGES_PER_WAITQUEUE 个队列所需的最小 2 的幂；其中 NoPages 是 zone 中的页数，PAGE_PER_WAITQUEUE 定义为 256。即：

wait_table_size = log2((NoPages * 2) / PAGES_PER_WAITQUEUE - 1)

字段 zone_t->wait_table_shift 是页地址右移多少位后可作为表内索引的位数。函数 page_waitqueue() 负责为某 zone 中的某页返回应使用的等待队列，使用基于 struct page 虚拟地址的简单乘法哈希算法。

具体做法是用地址乘以 GOLDEN_RATIO_PRIME，再将结果右移 zone_t->wait_table_shift 位作为哈希表索引。GOLDEN_RATIO_PRIME [Lev00] 是最接近该架构最大整数的黄金比率（golden ratio）[Knu68] 的最大质数。

2.3　Zone 的初始化

在 paging_init() 完成内核页表的全部设置之后才会初始化 zone。页表初始化将在 3.6 节介绍。可以预料，各架构以不同方式完成此任务，但目标始终一致：决定将哪些参数传给 free_area_init()（UMA）或 free_area_init_node()（NUMA）。UMA 只需要一个参数 zones_size。完整的参数列表如下：

  • nid　结点 ID，是被初始化结点的逻辑标识；
  • pgdat　被初始化结点的 pg_data_t。在 UMA 中即 contig_page_data；
  • pmap　稍后由 free_area_init_core() 设置，指向为该结点分配的局部 lmem_map 数组的起始位置。在 NUMA 中被忽略，因为 NUMA 将 mem_map 视为从 PAGE_OFFSET 开始的虚拟数组。在 UMA 中，该指针就是全局变量 mem_map，至此完成 UMA 下 mem_map 的初始化；
  • zones_sizes　包含每个 zone 页数大小的数组；
  • zone_start_paddr　第一个 zone 的起始物理地址；
  • zone_holes　包含各 zone 中内存空洞总大小的数组。

核心函数 free_area_init_core() 负责用相关信息填充每个 zone_t，并为该结点分配 mem_map 数组。注意，此时并不确定 zone 中哪些页是空闲的，这要到启动内存分配器退役时才知道（详见第 5 章）。

2.3.1　初始化 mem_map

mem_map 区域在系统启动期间以两种方式之一创建。在 NUMA 系统上，全局 mem_map 被视为从 PAGE_OFFSET 开始的虚拟数组；对每个活动结点调用 free_area_init_node()，为其分配该数组中属于自己的部分。在 UMA 系统上，free_area_init() 使用 contig_page_data 作为结点，全局 mem_map 同时充当该结点的"局部" mem_map。两个函数的调用图如图 2.5 所示。

图 2.5：函数调用图：free_area_init()

核心函数 free_area_init_core() 为正在初始化的结点分配一个局部 lmem_map。该数组的内存通过 alloc_bootmem_node() 由启动内存分配器分配（见第 5 章）。在 UMA 架构下，这块新分配的内存就成为全局的 mem_map；而 NUMA 稍有不同。

NUMA 架构在各自结点内分配 lmem_map 的内存。全局 mem_map 从未被显式分配，而是被置为 PAGE_OFFSET，作为虚拟数组对待。局部映射的地址保存在 pg_data_t->node_mem_map 中，位于虚拟 mem_map 内部。对于结点中存在的每个 zone，其在虚拟 mem_map 中的地址保存在 zone_t->zone_mem_map 中。其余代码都将 mem_map 当作真正的数组来使用，因为各结点只会使用其中有效的区域。

2.4　页（Pages）

系统中每个物理页帧都对应一个 struct page，用于跟踪其状态。在 2.2 内核 [BC00] 中，该结构与 System V [GC94] 中的对应结构很相似，但和其它 UNIX 变体一样，这个结构经历了较大的变化。在 <linux/mm.h> 中声明如下：

152 typedef struct page {
153     struct list_head list;
154     struct address_space *mapping;
155     unsigned long index;
156     struct page *next_hash;
158     atomic_t count;
159     unsigned long flags;
161     struct list_head lru;
163     struct page **pprev_hash;
164     struct buffer_head * buffers;
175
176 #if defined(CONFIG_HIGHMEM) || defined(WANT_PAGE_VIRTUAL)
177     void *virtual;
179 #endif /* CONFIG_HIGMEM || WANT_PAGE_VIRTUAL */
180 } mem_map_t;

各字段简要说明：

  • list　一个页可能属于多个链表，该字段用作链表头。例如，属于某个映射的页会挂在 address_space 维护的三个循环链表之一：clean_pages、dirty_pages 或 locked_pages。在 slab 分配器中，该字段用于保存指向该页所属 slab 和 cache 的指针。它也用于把空闲页块链接在一起；
  • mapping　当文件或设备被内存映射时，其 inode 拥有一个相关联的 address_space。若该页属于该文件，则此字段指向该地址空间。如果页是匿名的，且 mapping 字段已置位，则 address_space 是 swapper_space，管理交换地址空间；
  • index　含义随页面状态而变化。若页是文件映射的一部分，它表示页在文件内的偏移；若页是 swap cache 的一部分，它是页在交换地址空间（swapper_space）中的偏移。另外，当一组页正被释放给某个进程时，__free_pages_ok() 会将被释放页块的 order（即 2 的幂次方个数）保存到 index 中；
  • next_hash　属于文件映射的页按 inode 和偏移进行哈希。该字段把哈希到同一桶的页链接在一起；
  • count　页的引用计数。若降到 0，可被释放；大于 0 表示被一个或多个进程使用，或被内核用于等待 IO 等场景；
  • flags　描述页状态的标志位，全部声明于 <linux/mm.h>，列于表 2.1。还定义了一组用于测试、清除、设置这些位的宏（见表 2.2）。其中较值得注意的是 SetPageUptodate()，若架构定义了 arch_set_page_uptodate()，会在置位前先调用它；
  • lru　对于页面替换策略，可能被换出的页要么挂在 page_alloc.c 中声明的 active_list，要么挂在 inactive_list 上，该字段是这些 LRU 链表的链表头。这两个链表详见第 10 章；
  • pprev_hash　与 next_hash 配对使用，使哈希链表成为双向链表；
  • buffers　若页关联了某块设备的缓冲区，该字段用于跟踪 buffer_head。被进程映射的匿名页若有 swap 文件作为后备存储，也可能有关联的 buffer_head。这是因为页必须按底层文件系统定义的块大小与后备存储同步；
  • virtual　通常只有 ZONE_NORMAL 中的页被内核直接映射。要访问 ZONE_HIGHMEM 中的页需使用 kmap() 将页映射到内核（详见第 9 章）。可映射的页数有限。被映射后，该字段就是其虚拟地址。

类型 mem_map_t 是 struct page 的 typedef，便于在 mem_map 数组中引用。

PG_lru页在 active_list 或 inactive_list 上时置位
PG_referenced若页被映射，并通过 mapping/index 哈希表访问过则置位，用于在 LRU 链表间移动页
PG_reserved用于永远不会被换出的页。启动内存分配器（第 5 章）会为系统启动期间分配的页置位。后来也用于标记空页或不存在的页
PG_slab标记页被 slab 分配器使用
PG_skip某些架构用它跳过没有物理内存对应的地址空间部分
PG_unused字面意义上没有被使用
PG_uptodate页无错误地从磁盘读入后置位

2.4.1　将页映射到 Zone

直到 2.4.18，struct page 通过 page->zone 字段保存它所在 zone 的引用。但即使是这样一个小指针，在数以千计的 struct page 上累积起来也会消耗大量内存，因此被认为浪费。较新的内核移除了 zone 字段，改为使用 page->flags 的高 ZONE_SHIFT 位（在 x86 上为 8 位）来确定页所属 zone。首先在 mm/page_alloc.c 中建立了一个 zone 表 zone_table：

33 zone_t *zone_table[MAX_NR_ZONES*MAX_NR_NODES];
34 EXPORT_SYMBOL(zone_table);

MAX_NR_ZONES 是结点中可能的 zone 数量上限（即 3）；MAX_NR_NODES 是结点数量上限。EXPORT_SYMBOL() 使 zone_table 对可加载模块可见。这张表被当作多维数组使用。在 free_area_init_core() 中，结点的所有页都被初始化。首先设置表项：

733           zone_table[nid * MAX_NR_ZONES + j] = zone;

其中 nid 是结点 ID，j 是 zone 索引，zone 是 zone_t 结构。然后对每个页调用 set_page_zone()：

788           set_page_zone(page, nid * MAX_NR_ZONES + j);

参数 page 是被设置 zone 的页。可见，zone_table 中的索引被保存到了 page 中。

2.5　高端内存（High Memory）

由于内核可用地址空间（ZONE_NORMAL）大小有限，内核支持"高端内存"的概念。在 32 位 x86 系统上，高端内存有两个阈值：4 GiB 和 64 GiB。4 GiB 上限与 32 位物理地址可寻址的内存量有关。要访问 1 GiB 到 4 GiB 之间的内存，内核使用 kmap() 临时将高端内存中的页映射到 ZONE_NORMAL。详见第 9 章。

64 GiB 的第二个上限与物理地址扩展（Physical Address Extension，PAE）有关。PAE 是 Intel 的发明，使 32 位系统能使用更多的 RAM。它为内存寻址额外提供 4 位，可寻址多达 236 字节（64 GiB）。

理论上 PAE 允许处理器寻址多达 64 GiB，但在实践中，Linux 进程仍无法使用这么多 RAM，因为虚拟地址空间仍然只有 4 GiB。一些用户试图用一个进程 malloc() 出全部 RAM，结果发现行不通，由此颇感失望。

其次，PAE 也不允许内核自身使用如此多的内存。用于描述每个页帧的 struct page 仍占 44 字节，并消耗 ZONE_NORMAL 中的内核虚拟地址空间。也就是说，描述 1 GiB 内存大约需要 11 MiB 内核内存。因此 16 GiB 物理内存约消耗 176 MiB，给 ZONE_NORMAL 带来很大压力。再加上诸如页表项（PTE）等使用 ZONE_NORMAL 的其它结构，即便很小，最坏情况下也需要约 16 MiB。因此 x86 上 Linux 可用物理内存的实际上限约为 16 GiB。如果需要更多内存，建议非常简单直接：购置一台 64 位机器。

2.6　2.6 内核中的新变化

结点（Nodes）

初看之下，物理内存的描述方式似乎没有太大改动，但这些看似细小的变化影响很广。结点描述符 pg_data_t 新增了若干字段：

  • node_start_pfn　替代 node_start_paddr 字段。唯一区别是新字段保存 PFN 而非物理地址。这一改动是因为 PAE 架构可寻址超过 32 位的内存，超过 4 GiB 起始的结点无法用旧字段表示；
  • kswapd_wait　kswapd 的新等待队列。2.4 中只有一个全局换页守护进程等待队列；2.6 中每个结点都有一个 kswapdN（N 是结点 ID），每个都有自己的等待队列。

node_size 字段已被移除并替换为两个字段，以承认结点中可能存在没有物理内存的"空洞"。

  • node_present_pages　结点中实际存在的物理页总数；
  • node_spanned_pages　结点所覆盖的总区域，包括其中存在的空洞。
Zone

Zone 在 2.6 中差异显著。首先它不再叫 zone_t，而是直接用 struct zone。第二个重要变化是 LRU 链表。第 10 章会看到，2.4 内核中有一个全局链表决定页面释放或换出顺序，2.6 把这些链表放到了 struct zone 中。相关字段有：

  • lru_lock　本 zone LRU 链表的自旋锁。2.4 中是全局锁 pagemap_lru_lock；
  • active_list　本 zone 的活跃链表。与第 10 章描述的相同，只是变为每 zone 而非全局；
  • inactive_list　本 zone 的非活跃链表，2.4 中是全局的；
  • refill_counter　一次扫描中从 active_list 移除的页数，仅在页面替换时关心；
  • nr_active　active_list 上的页数；
  • nr_inactive　inactive_list 上的页数；
  • all_unreclaimable　若换页守护进程已两遍扫描完该 zone 全部页面仍无法释放足够内存，则置 1；
  • pages_scanned　自上次批量回收以来扫描过的页数。2.6 中页面是成组释放的，而不像 2.4 一次释放一个；
  • pressure　衡量该 zone 的扫描强度，是一种衰减平均值，影响扫描器回收页面的努力程度。

另外三个新字段与 zone 大小相关：

  • zone_start_pfn　zone 的起始 PFN，替代 2.4 中的 zone_start_paddr 与 zone_start_mapnr；
  • spanned_pages　zone 跨越的页数，包括某些架构中存在的空洞；
  • present_pages　zone 中实际存在的页数。许多架构上与 spanned_pages 相同。

另一个新增项是 struct per_cpu_pageset，用于为每个 CPU 维护页面链表，以减少自旋锁竞争。zone->pageset 是大小为 NR_CPU 的 struct per_cpu_pageset 数组，NR_CPU 是编译期的 CPU 数量上限。该 per-cpu 结构将在本节末讨论。

最后，struct zone 中新增了零填充。2.6 VM 的开发认识到某些自旋锁竞争非常激烈、获取非常频繁，且某些锁几乎总是成对获取，应尽量让它们位于不同的 cache line 上——这是常见的 cache 编程技巧 [Sea00]。这些填充使用 ZONE_PADDING() 宏标记，确保 zone->lock、zone->lru_lock 和 zone->pageset 字段使用不同的 cache line。

Pages

第一个显著变化是字段顺序的调整，使关联项尽量落到同一 cache line 中。字段本身基本相同，但新增了两项。第一是一个新 union，用于构建 PTE 链；PTE 链与页表管理相关，将在第 3 章末讨论。第二是 page->private 字段，包含与映射相关的私有信息。例如，若该页是缓冲区页，该字段保存指向 buffer_head 的指针。也因此移除了 page->buffers 字段。最后一个重要变化是：page->virtual 对高端内存支持不再必需，仅当架构明确要求时存在。高端内存页的支持详见第 9 章。

每 CPU 页面链表（Per-CPU Page Lists）

2.4 中只有 slab 分配器（见第 8 章）这一子系统主动维护对象的 per-cpu 链表。2.6 中此概念被广泛使用，并正式引入了"热页（hot）"和"冷页（cold）"的概念。

声明于 <linux/mmzone.h> 中的 struct per_cpu_pageset 只有一个字段，是一个含两个元素、类型为 per_cpu_pages 的数组。第 0 个元素用于热页，第 1 个用于冷页。"冷热"由该页当前在 cache 中的活跃程度决定。若已知页面短期内不会被引用（例如 IO 预读），就作为冷页分配。

struct per_cpu_pages 维护链表当前的页数、决定批量补充或释放时机的高/低水位、单次分配的页块大小，以及实际的页面链表头。

在 per-cpu 页面链表之上还有 per-cpu 页面计数机制。struct page_state 保存若干计数变量，例如跟踪本 CPU 已分配页数的 pgalloc 字段以及跟踪 swap 读入次数的 pswpin。该结构在 <linux/page-flags.h> 中有详细注释。提供 mod_page_state() 用于更新当前 CPU 的 page_state 字段，以及三个辅助宏 inc_page_state()、dec_page_state() 和 sub_page_state()。
