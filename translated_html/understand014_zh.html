第十一章　交换管理

原文版本：Chapter 11 — Swap Management

正如 Linux 会把空闲内存用作磁盘数据的缓冲区一样，最终也必须释放进程使用的私有页或匿名页。这类页与有磁盘文件作为后备的页不同，不能被简单丢弃然后稍后再读回来；而是要被仔细地复制到后备存储（backing storage），有时也称为交换区（swap area）。本章详细讨论 Linux 如何使用和管理它的后备存储。

严格地说，Linux 并不进行“交换（swapping）”，因为“swapping”一词指的是把进程的整个地址空间复制到磁盘，而“分页（paging）”才是指把单个页面写出。事实上，Linux 按现代硬件支持的方式实现的是分页，但在讨论和文档中传统上仍把它叫做 swapping。为与 Linux 中的惯用法保持一致，本书也使用“交换”一词。

存在交换空间是可取的，主要有两点理由。第一，它扩展了进程可使用的内存量。虚拟内存与交换空间使得即便进程只部分驻留内存，大型进程也能运行。由于“旧”页可被换出，被寻址的内存量可以轻易超过 RAM 大小，因为按需分页机制会在需要时把页面重新装回内存。

随便看一眼的读者1可能会认为，只要内存足够多就不需要交换；但这就引出了第二个理由。一个进程在生命周期早期引用的相当一部分页可能只用于初始化，之后再也不会用到。把这些页换出从而腾出更多磁盘缓冲区，比让它们继续驻留而无人使用更划算。

需要注意的是，交换并非没有缺点，其中最重要也最显而易见的一点就是：磁盘很慢，非常非常慢。如果若干进程在频繁地访问大量内存，再多的交换空间或者再昂贵的高性能磁盘也无法让它在合理时间内运行——只有增加 RAM 才能解决问题。这就是为什么如第 10 章所讨论的那样，换出“正确”的页非常重要；同时也需要把相关的页紧挨着保存在交换空间里，使它们在预读时有较大概率被一同读回。下面我们就从 Linux 如何描述一个交换区谈起。

本章开头会描述 Linux 为系统中每一个活动交换区所维护的数据结构，以及交换区在磁盘上的信息组织。然后讨论 Linux 在分页之后如何记住在交换区中找到这些页的方法，以及交换槽是如何被分配的。之后会讨论交换缓存（Swap Cache），它对共享页非常重要。到这里，信息已经足够让我们开始理解交换区的激活与停用、页的换入与换出、以及最终对交换区的读写过程。

11.1　描述交换区

每个活动的交换区，无论是文件还是分区，都有一个 swap_info_struct 结构来描述它。运行中系统里所有这样的结构都保存在一个静态声明的数组 swap_info 中，该数组拥有 MAX_SWAPFILES 个表项；MAX_SWAPFILES 被静态定义为 32，这意味着一个运行中的系统最多可存在 32 个交换区。swap_info_struct 在 <linux/swap.h> 中声明如下： 

 64 struct swap_info_struct {
 65     unsigned int flags;
 66     kdev_t swap_device;
 67     spinlock_t sdev_lock;
 68     struct dentry * swap_file;
 69     struct vfsmount *swap_vfsmnt;
 70     unsigned short * swap_map;
 71     unsigned int lowest_bit;
 72     unsigned int highest_bit;
 73     unsigned int cluster_next;
 74     unsigned int cluster_nr;
 75     int prio;
 76     int pages;
 77     unsigned long max;
 78     int next;
 79 };

下面对这个相当庞大的结构里每个字段作简要说明。

flags
这是一个位域，有两个可能的取值。如果交换区当前处于活动状态，就会置上 SWP_USED。SWP_WRITEOK 定义为 3，即最低两位，包含 SWP_USED 位。当 Linux 准备好向该区写入时（写入要求该区必须是活动的），flags 会被设为 SWP_WRITEOK。
swap_device
该交换区所用分区对应的设备保存在此。若交换区是一个文件，则该字段为 NULL。
sdev_lock
与 Linux 中许多结构一样，这个结构也需要保护。sdev_lock 是一个自旋锁，用于保护本结构，特别是 swap_map。可用 swap_device_lock() 与 swap_device_unlock() 来加锁和解锁。
swap_file
这是被挂载为交换区的实际特殊文件所对应的 dentry。例如，当一个分区被挂载用作交换区时，它可以是 /dev/ 目录下某个文件的 dentry。停用一个交换区时需要靠它来定位正确的 swap_info_struct。
vfs_mount
这是与该交换区设备或文件所在挂载点相对应的 vfs_mount 对象。
swap_map
这是一个大数组，交换区里每个交换条目（即每个页大小的槽位）都有一项。每一项是该槽位用户数的引用计数。交换缓存算作一个用户，每一个已被换出到该槽位的 PTE 也算作一个用户。如果该项等于 SWAP_MAP_MAX，则该槽位被永久分配；如果等于 SWAP_MAP_BAD，则该槽位永远不会被使用。
lowest_bit
这是该交换区中可能可用的最低空闲槽位号，用作线性扫描的起点以减少搜索空间。已知这个位置以下肯定不会有空闲槽。
highest_bit
这是该交换区中可能可用的最高空闲槽位号。与 lowest_bit 类似，已知此位置以上肯定不会有空闲槽。
cluster_next
下一个将要使用的块簇的偏移。交换区会尽量按簇分配，以增加相关页面被一起保存的概率。
cluster_nr
当前簇中仍剩余可分配的页数。
prio
每个交换区都有一个优先级，存储在此字段中。各区按优先级排序，并据此决定被使用的可能性。默认情况下，优先级按激活顺序排列，但管理员也可在使用 swapon 时通过 -p 选项指定。
pages
由于交换文件中部分槽位可能不可用，此字段保存交换区中可用页的数量。它与 max 的不同之处在于：标记为 SWAP_MAP_BAD 的槽位不计入其中。
max
该交换区中槽位的总数。
next
系统中下一个交换区在 swap_info 数组中的下标。

这些交换区虽然存储在数组里，但也被组织在一个伪链表 swap_list 中，其类型非常简单，在 <linux/swap.h> 中声明如下： 

153 struct swap_list_t {
154     int head;    /* head of priority-ordered swapfile list */
155     int next;    /* swapfile to be used next */
156 };

字段 swap_list_t→head 是当前使用中优先级最高的交换区，而 swap_list_t→next 是下一个应被使用的交换区。这样在寻找合适交换区时可以按优先级顺序排列各区，同时在必要时仍可通过数组进行快速查找。

每个交换区在磁盘上都被划分为若干页大小的槽位——例如在 x86 上每个槽就是 4096 字节。第一个槽位总是被保留，因为其中包含的关于交换区的信息不能被覆盖。交换区前 1 KiB 用于存放分区的磁盘标签，可被用户态工具读取。剩余空间则用于存放关于交换区的信息，由系统程序 mkswap 在创建交换区时填充。这些信息用于填充一个 union swap_header，其在 <linux/swap.h> 中声明如下： 

 25 union swap_header {
 26     struct
 27     {
 28         char reserved[PAGE_SIZE - 10];
 29         char magic[10];
 30     } magic;
 31     struct
 32     {
 33         char     bootbits[1024];
 34         unsigned int version;
 35         unsigned int last_page;
 36         unsigned int nr_badpages;
 37         unsigned int padding[125];
 38         unsigned int badpages[1];
 39     } info;
 40 };

下面对各字段进行说明。

magic
联合体中的 magic 部分仅用于识别“魔数”字符串。该字符串的存在是为了确保非交换区分区不会被误用，并据此判断交换区的版本。若字符串为“SWAP-SPACE”，则为版本 1 的交换文件格式；若为“SWAPSPACE2”，则为版本 2。其中那个很大的 reserved 数组只是为了让魔数字符串恰好位于这一页的末尾被读取。
bootbits
这是预留区域，包含关于分区的信息（例如磁盘标签）。
version
交换区布局的版本号。
last_page
交换区中最后一个可用页。
nr_badpages
本字段保存交换区中已知的坏页数量。
padding
磁盘扇区通常约为 512 字节。version、last_page 和 nr_badpages 三个字段占 12 字节，padding 用于填充剩余的 500 字节，正好凑满一个扇区。
badpages
该页的剩余空间用于保存最多 MAX_SWAP_BADPAGES 个坏槽位的下标。这些槽位由 mkswap 在使用 -c 选项检查交换区时填入。

MAX_SWAP_BADPAGES 是一个编译期常量，随结构变化而变化，但按其当前形式由下面这个简单等式可得为 637： 

MAX_SWAP_BADPAGES = (PAGE_SIZE - 1024 - 512 - 10) / sizeof(long)

其中 1024 是引导块大小，512 是 padding 大小，10 是用于标识交换文件格式的魔数字符串长度。

11.2　把页表项映射为交换条目

当一页被换出时，Linux 使用对应的 PTE 来保存足够信息，以便日后在磁盘上重新定位该页。显然，PTE 本身大小不足以保存该页在磁盘上的精确位置，但用来保存 swap_info 数组中的下标和该页在 swap_map 中的偏移则绰绰有余——而这正是 Linux 所做的。

无论是哪种体系结构，每个 PTE 都足够大，可以存下一个 swp_entry_t。其在 <linux/shmem_fs.h> 中声明如下： 

 16 typedef struct {
 17     unsigned long val;
 18 } swp_entry_t;

有两个宏分别用于在 PTE 与交换条目之间互相转换：pte_to_swp_entry() 和 swp_entry_to_pte()。

每种体系结构都必须能够判断一个 PTE 是处于“present”状态还是已被换出。下面我们以 x86 为例说明其实现方式。在 swp_entry_t 中始终保留两位：x86 上的第 0 位用于 _PAGE_PRESENT 标志，第 7 位用于 _PAGE_PROTNONE。这两位的必要性已在第 3.2 节中说明。第 1–6 位用于type，即在 swap_info 数组中的下标，由 SWP_TYPE() 宏返回。

第 8–31 位用于在 swp_entry_t 中存放 swap_map 内的offset。在 x86 上，这意味着有 24 位可用，“把”交换区大小“限制”在 64 GiB。SWP_OFFSET() 宏用于提取该偏移。

为了把 type 与 offset 编码到 swp_entry_t 中，可使用 SWP_ENTRY() 宏，它只是执行相应的位移操作。这些宏之间的关系如图 11.1所示。

图 11.1：在 swp_entry_t 中存储交换条目信息

应当注意，“type”占用的 6 位本来允许 32 位体系结构上存在多达 64 个交换区，而不是被限制为 MAX_SWAPFILES（即 32）。这个限制源自 vmalloc 地址空间的消耗。如果一个交换区取最大可能尺寸，则 swap_map 需要 32 MiB（224 * sizeof(short)）——记住每页占用一个 short 用作引用计数。哪怕仅仅是 MAX_SWAPFILES 个最大尺寸的交换区，也要 1 GiB 的虚拟 malloc 空间，鉴于用户/内核线性地址空间的划分，这是根本不可能的。

这看起来似乎说明：支持 64 个交换区并不值得增加这点复杂度。然而，即使总交换容量并不增加，也存在希望拥有大量交换区的情形。一些现代机器2有多个独立磁盘，可在它们之间产生大量独立块设备。这种情况下，希望创建大量小交换区均匀分布在所有磁盘上。这样可以在分页行为上获得高度的并行性，这对交换密集型应用很重要。

11.3　分配一个交换槽位

所有页大小的槽位都由数组 swap_info_struct→swap_map（类型为 unsigned short）追踪。每一项都是该槽位使用者数量的引用计数（在共享页的情形下会大于 1），为 0 表示空闲。如果某项等于 SWAP_MAP_MAX，则该页被永久保留给该槽。这种情形即便不是不可能，也极不可能发生，存在该值是为了避免引用计数溢出。如果该项等于 SWAP_MAP_BAD，则该槽不可用。

图 11.2：调用图：get_swap_page()

查找并分配一个交换条目的任务分为两个主要步骤。第一步由高层函数 get_swap_page() 完成。它从 swap_list→next 开始，在各交换区中搜索合适的槽位。找到后记录下一次应使用的交换区，并返回所分配的条目。

搜索 swap_map 的工作由 scan_swap_map() 负责。原则上很简单——线性扫描数组找到一个空闲槽位然后返回；可以预见，实际实现要细致一些。

Linux 试图把页在磁盘上组织成大小为 SWAPFILE_CLUSTER 的簇（cluster）。它在交换空间中连续分配 SWAPFILE_CLUSTER 个页，并在 swap_info_struct→cluster_nr 中记录连续分配的页数，在 swap_info_struct→cluster_next 中记录当前偏移。一旦一段连续块分配完毕，就再寻找 SWAPFILE_CLUSTER 大小的空闲段；若找到足够大的块，便作为下一个簇大小的连续序列使用。

若在交换区中找不到足够大的空闲簇，则从 swap_info_struct→lowest_bit 开始执行简单的首次适应（first-free）搜索。其目标是让同时换出的页在磁盘上挨得近，前提是同时换出的页彼此相关。乍一看这个前提似乎奇怪，但考虑到页置换算法在交换空间使用最频繁时正是线性扫描进程地址空间换出页，这个前提就相当可靠了。如果不去寻找大块空闲段并加以利用，那么扫描就会退化为单纯的首次适应搜索，永远得不到改善。借助这种机制，进程退出时往往会释放出大块连续槽位。

11.4　交换缓存

在多个进程之间共享的页很难简单地换出，因为如前所述，没有快速的方法把一个 struct page 映射回每个引用它的 PTE。这就会产生竞争条件：一个 PTE 中页仍然存在，而另一个 PTE 中页已被换出，更新后未同步到磁盘从而丢失更新。

为解决该问题，那些在后备存储中已预留有槽位的共享页被视为交换缓存（swap cache）的一部分。交换缓存纯粹是概念性的，它实际上只是页缓存的一种特化。交换缓存中页与普通页缓存中页的首要差别是：交换缓存中页始终使用 swapper_space 作为其 page→mapping 中的 address_space。第二个差别是：页被加入交换缓存使用 add_to_swap_cache() 而不是 add_to_page_cache()。

图 11.3：调用图：add_to_swap_cache()

匿名页在试图被换出之前并不属于交换缓存。变量 swapper_space 在 swap_state.c 中声明如下： 

 39 struct address_space swapper_space = {
 40     LIST_HEAD_INIT(swapper_space.clean_pages),
 41     LIST_HEAD_INIT(swapper_space.dirty_pages),
 42     LIST_HEAD_INIT(swapper_space.locked_pages),
 43     0,
 44     &swap_aops,
 45 };

一旦 page→mapping 字段被设为 swapper_space，该页就被认为是交换缓存的一部分，这一点由 PageSwapCache() 宏来测试。Linux 用同一套代码维护“交换与内存之间”的同步以及“文件后备页与内存之间”的同步，二者共享页缓存代码，区别仅在于所用的具体函数。

作为后备存储的地址空间 swapper_space，其 address_space→a_ops 使用 swap_ops。page→index 字段则不再像通常那样保存文件偏移，而是用于保存 swp_entry_t 结构。address_space_operations 结构 swap_aops 在 swap_state.c 中声明如下： 

 34 static struct address_space_operations swap_aops = {
 35     writepage: swap_writepage,
 36     sync_page: block_sync_page,
 37 };

当一页被加入交换缓存时，先通过 get_swap_page() 分配一个槽位，再通过 add_to_swap_cache() 加入页缓存，然后标记为 dirty。下次该页被“洗涤（laundered）”时，会按照普通页缓存的方式被真正写到磁盘后备存储。整个过程如图 11.4 所示。

图 11.4：把页加入交换缓存

后续若该页被从其他共享 PTE 中再次换出，会调用 swap_duplicate()，它只是把 swap_map 中对应槽位的引用计数加一。若该 PTE 因写入被硬件标记为 dirty，该位会被清除，并通过 set_page_dirty() 把 struct page 标记为 dirty，从而保证在页被丢弃前磁盘副本会被同步。这样可以确保：在对该页的所有引用都被释放之前，会进行检查以保证磁盘上的数据与页帧中的数据一致。

当对该页的引用计数最终降为 0 时，该页就可以从页缓存中被丢弃；此时 swap_map 中的计数仍记录着磁盘槽位归属的 PTE 数量，因此该槽位不会被过早释放。该页随后按第 10 章所述的同一套 LRU 老化和逻辑被洗涤并最终丢弃。

反过来，如果对一个“已换出”页发生了缺页，do_swap_page() 中的逻辑会调用 lookup_swap_cache() 检查该页是否存在于交换缓存中。如果存在，PTE 会被更新为指向该页帧，页引用计数加一，并通过 swap_free() 把交换槽位计数减一。

swp_entry_t get_swap_page()
该函数通过搜索活动交换区，在某个 swap_map 中分配一个槽位。详细内容见第 11.3 节。此处列出是因为它主要与交换缓存配合使用。
 
int add_to_swap_cache(struct page *page, swp_entry_t entry)
该函数把一页加入交换缓存。它首先调用 swap_duplicate() 检查页是否已存在；若不存在，则通过普通的页缓存接口函数 add_to_page_cache_unique() 把它加入交换缓存。
 
struct page * lookup_swap_cache(swp_entry_t entry)
该函数搜索交换缓存并返回与给定 entry 对应的 struct page。它的工作方式是基于 swapper_space 和 swap_map 偏移在普通页缓存中进行搜索。
 
int swap_duplicate(swp_entry_t entry)
该函数校验一个交换条目是否有效，若有效则把其 swap_map 计数加一。
 
void swap_free(swp_entry_t entry)
swap_duplicate() 的对偶函数。它把 swap_map 中相应的计数减一。当计数降为 0 时，槽位实际上就空闲了。

11.5　从后备存储读取页

在读入页时主要使用的函数是 read_swap_cache_async()，它主要在缺页处理过程中被调用。该函数首先用 find_get_page() 搜索交换缓存。通常对交换缓存的搜索由 lookup_swap_cache() 完成，但该函数会更新搜索次数统计；而由于缓存可能需要被多次搜索，这里改用 find_get_page()。

图 11.5：调用图：read_swap_cache_async()

若另一个进程也映射了同一页，或者多个进程同时对同一页缺页，则该页可能已经存在于交换缓存中。如果不在交换缓存中，则必须分配一页并用后备存储中的数据填充。

页通过 alloc_page() 分配之后，会用 add_to_swap_cache() 加入交换缓存，因为交换缓存操作只能针对位于交换缓存中的页进行。若无法加入交换缓存，则再次搜索交换缓存以确认是否已有别的进程把数据放入了其中。

为从后备存储读入信息，会调用 rw_swap_page()，详见第 11.7 节。函数完成后，会调用 page_cache_release() 释放 find_get_page() 对该页的引用。

11.6　把页写入后备存储

当任何一页被写到磁盘时，系统会查询 address_space→a_ops 以找到合适的写出函数。对后备存储而言，address_space 是 swapper_space，相关的交换操作包含在 swap_aops 中。结构 swap_aops 将 swap_writepage() 注册为其写出函数。

图 11.6：调用图：sys_writepage()

swap_writepage() 的行为取决于执行写入的进程是否是该交换缓存页的最后一个用户。它通过调用 remove_exclusive_swap_page() 判断这一点——后者在持有 pagecache_lock 的情况下检查页的计数。如果没有别的进程映射该页，则将其从交换缓存中移除并释放。

如果 remove_exclusive_swap_page() 已经把页从交换缓存中移除并释放，swap_writepage() 会解锁该页，因为它已不再被使用。如果它仍存在于交换缓存中，则调用 rw_swap_page() 把数据写入后备存储。

11.7　对交换区进行读写

对交换区进行读写的顶层函数是 rw_swap_page()。该函数确保所有操作都经过交换缓存，以防止丢失更新。rw_swap_page_base() 是真正执行工作的核心函数。

它首先判断操作是否为读取。若是，则用 ClearPageUptodate() 清除 uptodate 标志——既然需要 I/O 来填充数据，页显然就不是最新的。读取成功之后，该标志会被再次置位。然后调用 get_swaphandle_info() 获取交换分区所属的设备，或者交换文件所属的 inode。这些信息是块层执行实际 I/O 所需要的。

核心函数对交换分区和交换文件都能工作，因为它使用块层函数 brw_page() 来完成真正的磁盘 I/O。如果交换区是一个文件，则使用 bmap() 用文件系统中所有包含该页数据的块号填充一个本地数组。要记得，文件系统可能有自己存放文件和磁盘数据的方式，没有交换分区那么简单——交换分区里信息可以直接写到磁盘。如果后备存储是分区，那么只需要对一个页大小的块执行 I/O；且由于不涉及文件系统，bmap() 也就没必要。

一旦确定了需要读写哪些块，就用 brw_page() 执行常规的块 I/O 操作。所有 I/O 都是异步的，因此函数会很快返回。一旦 I/O 完成，块层会解锁该页，所有等待该页的进程都会被唤醒。

11.8　激活一个交换区

现在我们已经讨论了交换区是什么、如何表示、页如何被追踪，是时候看看它们是如何串起来以激活一个交换区的。激活一个交换区在概念上相当简单：打开文件、从磁盘读入头信息、填充一个 swap_info_struct、并把它加入交换链表。

负责激活交换区的函数是 sys_swapon()，它接受两个参数：交换区特殊文件的路径，以及一组标志。在交换区被激活期间，大内核锁（Big Kernel Lock，BKL）会被持有，这阻止任何应用程序在此操作进行期间进入内核态。该函数相当庞大，但可分解为如下几个简单步骤：

  • 在 swap_info 数组中找到一个空闲的 swap_info_struct，并用默认值初始化它；
  • 调用 user_path_walk()，它会沿目录树遍历用户给定的 specialfile，并用关于该文件可用的数据（例如 dentry 以及文件所在文件系统的信息 vfsmount）填充一个 namidata 结构；
  • 填充 swap_info_struct 中与交换区尺寸和寻址方式相关的字段。若交换区是一个分区，会先把块大小设置为 PAGE_SIZE 再计算大小；若是文件，则信息直接从 inode 中获得；
  • 确保该区尚未被激活。如未被激活，则分配一页内存并读入交换区的第一个页大小的槽位。该页包含若干信息，例如可用槽位数，以及如何用坏槽位填充 swap_info_struct→swap_map 的依据；
  • 用 vmalloc() 为 swap_info_struct→swap_map 分配内存，并把好槽位初始化为 0，其他槽位初始化为 SWAP_MAP_BAD。理想情况下头信息应为版本 2 的文件格式，因为版本 1 在 x86 这种 4 KiB 页大小的体系结构上把交换区大小限制在不到 128 MiB3；
  • 在确认头中信息与实际交换区一致后，填好 swap_info_struct 中其余的信息，比如最大页数和可用好页数。更新全局统计变量 nr_swap_pages 和 total_swap_pages；
  • 此时交换区已完全激活并初始化完成，根据其优先级把它插入到交换链表中正确的位置。

函数结束时释放 BKL，系统就有了一个可供分页使用的新交换区。

11.9　停用一个交换区

相比激活而言，停用交换区的开销极其高昂。主要问题在于：交换区不能简单地直接被移除，每个被换出的页都必须再次换入。正如没有快速方法把一个 struct page 映射回每一个引用它的 PTE 一样，也没有快速方法把一个交换条目映射回 PTE。这就要求遍历所有进程页表，找出引用要停用交换区的所有 PTE 并把它们换入。这当然也意味着：如果物理内存不足，停用就会失败。

不出所料，负责停用一个交换区的函数叫做 sys_swapoff()。该函数主要关心更新 swap_info_struct。把每个已换出页换入的繁重任务则交给 try_to_unuse()，其代价极为高昂。对于 swap_map 中使用着的每一个槽位，都得遍历各进程的页表去寻找它。在最坏情况下，所有 mm_struct 所属的页表都得遍历。因此，停用一个交换区大致需要执行如下任务：

  • 调用 user_path_walk() 获取要停用的特殊文件的信息，然后取得 BKL；
  • 从交换链表中移除该 swap_info_struct，更新全局统计：可用交换页数（nr_swap_pages）和总交换条目数（total_swap_pages）。一旦完成，就可以释放 BKL；
  • 调用 try_to_unuse()，它会把要停用交换区中的所有页换入。该函数通过 find_next_to_unuse() 在 swap_map 中循环查找下一个被使用的交换槽位。对每个已用槽位，执行如下操作：
    • 调用 read_swap_cache_async() 为磁盘上保存的槽位分配一页。理想情况下它已经在交换缓存中，否则会调用页分配器；
    • 等待该页被完全换入并锁定。锁定后，对每个引用该页的进程都调用 unuse_process()。该函数遍历页表搜索相关 PTE，并将其更新为指向该 struct page。若该页是已无其他引用的共享内存页，则改为调用 shmem_unuse()；
    • 释放所有曾经被永久映射的槽位。这里假设槽位不会真的变为永久保留，所以承担这点风险；
    • 把页从交换缓存中删除，防止 try_to_swap_out() 在该页仍可能于 swap_map 中存在引用的情况下再次去引用它。
  • 如果可用内存不足以换入全部条目，由于不能简单丢弃该交换区，就会把它重新插回到运行系统中。如果成功，则把 swap_info_struct 置为未初始化状态，并通过 vfree() 释放 swap_map 内存。

11.10　2.6 中的新变化

对 struct swap_info_struct 最重要的新增是加入了一个名为 extent_list 的链表，以及一个名为 curr_swap_extent 的缓存字段，用于实现“扩展段（extent）”。

扩展段由 struct swap_extent 表示，把交换区中连续的一段页映射到磁盘上连续的一段块。这些扩展段在 swapon 时由 setup_swap_extents() 建立。对块设备而言，只会有一个交换扩展段，并不会带来性能提升；但仍然建立扩展段，以便后备为块设备或普通文件的交换区可以被同等对待。

对交换文件来说，差别可能非常大——它会包含多个扩展段，每个段代表一组按块聚集在一起的页范围。当搜索某一偏移上的页时，会遍历扩展段链表。为加快搜索速度，最近一次被搜索的扩展段会缓存在 swap_extent→curr_swap_extent 中。
